# 阶段 6：插件创建与 StateGraph 在线编辑器

> 支持从现有 Skill（SKILL.md）一键生成插件的 scenario 与状态机草稿，用户可在前端通过图形化或 YAML 编辑器交互调整后发布；同时支持对已有插件的状态机进行在线修改；插件采用 Draft / 正式版本两态管理，编辑中的变更以 Draft 形式持久保存，发布后生成正式版本并即时热重载生效，Chat 对话默认绑定最新发布版本，后续修改不影响历史对话。

---

## 一、权限与访问控制

1. **管理员限定**：StateGraph 编辑器仅对具有管理员角色的用户开放；在开发调试环境下可通过配置允许非管理员访问。
2. **只读模式降级**：普通用户访问时，StateGraph 仅以只读模式展示（同阶段 5），不显示编辑入口。

## 二、状态机读写接口

3. **读取接口**：提供接口返回插件当前状态机定义，包含：图结构（节点、边及其属性）、节点在画布上的位置布局、原始 YAML 文本；可通过参数指定读取 Draft 或某个正式版本。
4. **Draft 保存接口**：编辑器中每次保存操作将修改持久化为该插件的 Draft 版本（覆盖上一次 Draft），不产生正式版本，不影响任何 Chat 会话；保存前执行合法性校验（详见第五节），校验失败返回详细错误列表。
5. **发布接口**：将当前 Draft 内容发布为新的正式版本（版本号自增，格式 `v1`、`v2` …），发布后 Draft 清空，运行中服务无需重启即可加载新版本状态机；新建 Chat 会话将默认绑定最新发布版本；已在运行中的 Chat 会话继续使用其启动时绑定的版本，不受影响。
6. **持久化存储**：Draft 与所有正式版本均按记录存入数据库，每条记录包含版本号（Draft 标记为 `draft`）、状态（`draft` / `published`）、创建时间、修改者及完整状态机内容；不依赖容器本地文件系统，保证重启和多副本部署下的一致性；原有文件作为初始导入来源，导入后以数据库为准，导入版本号为 `v1`。

## 三、状态机合法性校验

7. **结构完整性**：状态机必须包含 `__start__` 和 `__end__` 节点；不允许存在孤立节点（无输入且无输出的节点，`__start__` 除外）；不允许存在环路。
8. **节点输入输出规则**：
   - 每个节点必须至少有一条输出边（`__end__` 节点除外）；
   - 除 `__start__` 节点外，每个节点必须至少有一条输入边；
   - 输出边需指定 `artifact-key`，表示该节点产出的制品标识；
   - 输入不指定来源节点，而是指定 `artifact-key`；输入所引用的 `artifact-key` 必须由拓扑上位于其前序的节点产出（不允许引用后继节点或平行分支的 `artifact-key`）。
9. **Artifact 一致性**：`artifact-key` 及其类型在整个状态机内统一定义（在插件级别集中声明，不在各节点中分散定义），多个节点可共用同一 `artifact-key`；校验时检查所有引用该 `artifact-key` 的节点所声明的类型是否与集中定义一致，不一致则报错。
10. **条件完整性**：转移边的 condition 字段不得为空。
11. **校验反馈**：Draft 保存及发布前均执行上述校验；校验失败时在编辑器内直接展示错误列表，标注出问题的节点/边，不关闭编辑器；用户修正后可再次保存或发布。

## 四、图形化编辑

12. **节点编辑**：支持新增、删除步骤节点；可编辑步骤 ID（唯一键）、显示标签、步骤默认 human/auto 模式；可在节点属性面板中配置该节点的输出 `artifact-key`（从插件级 artifact 定义中选择或新建）及输入所依赖的 `artifact-key`（从前序节点产出的列表中选择）。
13. **边编辑**：支持在节点间连线新增转移边、删除已有边；可内联编辑转移条件文本。
14. **布局持久化**：节点拖拽后的位置随 Draft 一同保存，下次打开编辑器时恢复上次布局；不同插件的布局独立存储。

## 五、YAML 双向编辑

15. **YAML 预览面板**：编辑器内提供 YAML 文本视图，图形编辑时实时同步更新对应 YAML 内容；用户可直接在 YAML 视图中修改，修改后图形视图自动更新。
16. **两种视图保持同步**：图形视图与 YAML 视图始终反映同一份状态机定义，切换视图时不丢失未保存的修改。

---

## 六、版本管理与发布

17. **版本状态**：每个插件同时维护一份 Draft 和若干正式版本；Draft 不可被 Chat 会话加载，仅供编辑预览；只有正式版本可供 Chat 使用。
18. **版本绑定**：Chat 会话在启动时绑定当前最新发布版本（或用户显式指定的版本），后续对话全程使用该版本，不随插件后续发布而变化。
19. **显式版本选择**：用户可在新建 Chat 时从已发布版本列表中显式选择特定版本，选定后该会话全程锁定该版本。
20. **版本列表接口**：提供接口返回指定插件的所有已发布版本列表，包含版本号、发布时间、发布者、简要备注（可选填）；Draft 单独通过读取接口获取，不出现在版本列表中。
21. **版本回滚**：管理员可将任意已发布历史版本重新设为"最新版本"（实质是基于该版本内容发布一个新版本号），后续新建 Chat 加载该版本；版本历史保持不可变。
22. **版本差异对比**：提供接口及前端 UI 支持对任意两个已发布版本（或 Draft 与某发布版本）进行 Diff 展示（节点/边的增删改）。
23. **版本清理策略**：可配置最大保留版本数（默认 50），超出后自动删除最旧版本，但仍被活跃 Chat 会话引用的版本不得删除。

---

## 七、从 Skill 生成插件（创建入口）

24. **Skill → 插件草稿**：用户可从本地已有 Skill 或通过 URL 引入网络上的 Skill（SKILL.md 格式），系统调用 AI 分析其工作流描述，自动生成对应的插件草稿，包含：`scenario.md`（意图识别描述）、`state.yml`（步骤列表与状态机转移）、`plugin.yaml`（插件元数据与 UI slot 声明）；生成结果以 Draft 形式保存，不自动发布。
25. **草稿预览与交互修改**：生成的草稿在前端编辑器中展示，用户可在发布前对步骤名称、转移条件、slot 配置等进行交互式调整（使用上方图形化编辑器或 YAML 编辑器）。
26. **发布为插件**：用户确认草稿后一键发布，触发发布接口，插件首个正式版本（`v1`）立即注册到系统并对新建 Chat 可用，无需部署操作。
27. **从零新建**：不依赖已有 Skill，支持用户直接在编辑器中从空白画布新建插件，手动定义步骤与状态机，保存为 Draft 后按需发布。
