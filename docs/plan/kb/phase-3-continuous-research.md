# 阶段三：持续研究与知识演进

## 1. 阶段目标

让知识探索从一次性聊天升级为持续运行的研究任务：知识发生变化时，Agent 能够增量分析、更新结论、发现冲突与知识缺口，并在受控流程中建议沉淀新知识。

本阶段解决“知识变化后怎么办”。

阶段二的 Exploration Plan、分支状态、证据充分度和引用关系必须已经结构化并可持久化。

## 2. 用户场景

首批覆盖：

1. 每日或每周监控知识库新增内容。
2. 新政策发布后生成变化摘要。
3. 判断已有制度问答是否过期。
4. 回答某个历史时间点有效的规定。
5. 查找并比较同一文档的多个版本。
6. 新证据影响已有结论时通知用户。
7. 自动维护 FAQ、主题地图和决策记录。
8. 定期生成知识缺口报告。
9. 对已保存的研究任务进行增量更新。
10. 将研究成果提交为待审批知识资产。

## 3. 产品交付

### 3.1 持续研究

用户可以把一次深入探索保存为持续研究，并设置：

- 关注的知识库和主题。
- 触发方式：知识变更事件或定时运行。
- 通知频率和变化阈值。
- 是否自动更新报告草稿。
- 是否需要人工确认新增研究分支。

### 3.2 变化视图

每次增量运行展示：

- 新增、修改、删除了哪些证据。
- 哪些旧结论仍成立。
- 哪些结论被增强、削弱、推翻或需要复核。
- 哪些引用已经失效或指向旧版本。
- 哪些过去的未知项得到补充。
- 是否产生新的冲突和研究分支。

### 3.3 知识资产维护

支持生成并维护：

- FAQ。
- 主题地图。
- 研究报告。
- 决策记录。
- 版本差异报告。
- 知识缺口清单。

Agent 只生成修改建议或草稿。发布、覆盖、归档和删除遵循既有权限、审批、Diff、回滚和审计机制。

## 4. 技术交付

### 4.1 文档版本和时态模型

为文档及其索引内容引入：

```json
{
  "document_id": "doc_xxx",
  "version_id": "ver_xxx",
  "previous_version_id": "ver_prev",
  "content_hash": "...",
  "valid_from": "...",
  "valid_to": null,
  "supersedes": ["ver_old"],
  "status": "effective",
  "change_summary": "..."
}
```

状态至少包括：

- `draft`
- `effective`
- `expired`
- `archived`

新增检索能力：

```text
search_knowledge_as_of(query, timestamp, scope)
find_document_versions(document_id)
compare_document_versions(left_version, right_version)
get_effective_document(document_id, timestamp)
```

检索结果和引用必须包含文档版本及有效时间。

### 4.2 知识变更事件

统一事件：

- `document_added`
- `document_updated`
- `document_deleted`
- `document_reparsed`
- `document_status_changed`
- `knowledge_space_changed`
- `data_source_synced`

事件至少携带：

- 知识库 ID。
- 文档和版本 ID。
- 事件时间。
- 旧、新内容 hash。
- 变更类型和摘要。
- 受影响的主题和实体。

事件应具有幂等 ID，消费者支持断点续传和失败重试。

### 4.3 研究任务模型

持续研究保存：

```json
{
  "research_id": "research_xxx",
  "goal": "持续跟踪公司差旅制度变化",
  "knowledge_policy": {},
  "exploration_plan_id": "plan_xxx",
  "trigger": {},
  "last_cursor": {},
  "last_run_at": "...",
  "notification_policy": {},
  "status": "active"
}
```

每次运行产生不可变 run 记录，保存输入事件、受影响分支、工具调用、变化结论和产物版本。

### 4.4 结论—证据依赖图

建立：

```text
持续研究
  → 研究计划
  → 研究分支
  → 结论
  → 证据片段
  → 文档版本
```

结论记录：

- 结论内容和类型。
- 直接支持证据。
- 反对或限制证据。
- 推导自哪些子结论。
- 置信度。
- 首次产生和最后验证时间。

文档变化后，从受影响的证据节点反向找到需要重跑的分支和结论。

### 4.5 增量研究

处理流程：

```text
知识变更事件
  → 识别受影响的主题、实体、证据和研究任务
  → 找到受影响分支
  → 只重跑相关路由和检索
  → 对比新旧证据及结论
  → 更新研究 run
  → 达到通知阈值时通知用户
```

以下情况允许完整重跑：

- 知识库画像或索引算法发生不兼容升级。
- 研究目标发生实质变化。
- 依赖图损坏或缺失。
- 大量文档批量替换，增量成本接近全量成本。

### 4.6 冲突和失效检测

新证据进入时检查：

- 是否与现有结论直接矛盾。
- 是否替代了旧版本。
- 是否使引用失效。
- 是否只改变局部适用范围。
- 是否补齐过去的未知项。
- 是否使原有概括不再具有足够样本支持。

变化类型：

- `supported`
- `strengthened`
- `weakened`
- `contradicted`
- `superseded`
- `scope_changed`
- `unresolved`

### 4.7 知识缺口

将以下情况归为知识缺口：

- 多次被询问但没有直接证据。
- 只有单一、过期或低质量来源。
- 不同知识库长期冲突且没有权威来源。
- 高优先级研究分支持续无法完成。
- 文档提到重要实体，但没有对应资料。
- 现有 FAQ 或报告频繁被用户纠正。

缺口报告包括：

- 缺失问题或主题。
- 业务影响和出现频率。
- 已搜索范围。
- 建议补充的资料类型和责任人方向。
- 是否适合创建 FAQ、制度、案例或数据源同步任务。

### 4.8 受控知识沉淀

Agent 可提出：

- 创建 FAQ 草稿。
- 更新研究报告。
- 生成版本变化说明。
- 创建新的专题知识文档。
- 建议归档或替换旧文档。

每项建议必须带：

- 来源研究和证据。
- 目标知识库。
- 新旧内容 Diff。
- 修改理由。
- 受影响引用或下游资产。
- 审批状态和审计记录。

## 5. 调度与通知

### 5.1 触发方式

- 事件触发：相关知识库发生变化。
- 定时触发：每日、每周、每月复查。
- 手动触发：用户要求立即刷新。
- 依赖触发：上游持续研究产出变化。

### 5.2 通知抑制

以下情况不通知：

- 只有格式、分段或解析变化，语义未变。
- 新证据与已有证据重复。
- 低优先级分支发生轻微变化。
- 同一批同步产生的重复事件。

以下情况优先通知：

- 核心结论被推翻或替代。
- 关键制度生效或失效。
- 高优先级未知项得到答案。
- 出现新的直接冲突。
- 已发布知识资产需要更新。

## 6. 实施任务

### 6.1 Data Model

- 增加文档版本、有效期、状态和替代关系。
- 为 chunk 和引用绑定文档版本。
- 建立持续研究、研究 run、结论和依赖关系模型。
- 为知识资产保存来源研究和证据链。

### 6.2 Event Pipeline

- 定义统一知识变更事件 schema。
- 在上传、同步、解析、删除和状态变化链路产生事件。
- 实现幂等消费、游标、重试和死信处理。
- 实现研究任务与事件的影响匹配。

### 6.3 Agent Runtime

- 增加时态和版本工具。
- 实现受影响分支识别和增量重跑。
- 实现新旧结论 Diff 和变化分类。
- 实现冲突、失效和知识缺口检测。
- 实现受控知识沉淀建议。

### 6.4 Frontend

- 支持保存、暂停、恢复和删除持续研究。
- 展示每次 run 的变化摘要。
- 展示结论状态和证据变化。
- 支持通知策略和研究范围配置。
- 提供知识资产 Diff、审批和发布入口。

### 6.5 Evaluation

- 建立文档版本和历史时间点评测集。
- 建立新增证据支持、削弱、推翻结论的样例。
- 测量受影响分支识别、过期引用和通知有效性。
- 对比增量执行与完整重跑的质量、成本和延迟。

## 7. 验收指标

- 文档变化影响分支识别准确率不低于 90%。
- 增量研究比完整重跑节省至少 70% 的检索和模型成本。
- 过期引用发现率不低于 95%。
- 版本和时间类问题答案准确率不低于 90%。
- 变化通知中的有效变化比例不低于 80%。
- 同一知识变更不产生重复研究 run 或重复通知。
- 自动生成知识资产的人工接受或轻量修改后接受比例不低于 60%。

## 8. 完成定义

- 文档版本、有效期和替代关系可被检索和引用。
- 所有核心知识变更链路可靠产生事件。
- 研究结论与证据存在可追踪依赖关系。
- 持续研究支持增量重跑、暂停、恢复和变化通知。
- 新旧结论变化可以被结构化解释。
- 知识沉淀具备证据、Diff、审批、回滚和审计能力。

## 9. 非目标

- 不允许 Agent 绕过审批直接覆盖正式知识资产。
- 不因为一次低置信度冲突自动判定旧结论错误。
- 不要求所有历史文档自动推断出精确有效期；无法确定时明确标记未知。
- 不把知识事件流扩展为通用业务事件平台。
