# 阶段一：Agent 自主找到知识

## 1. 阶段目标

即使用户没有选择知识库，Agent 也能稳定完成：

```text
理解问题
  → 发现候选知识库
  → 选择检索范围
  → 获取证据
  → 判断相关性和充分度
  → 必要时改写 query 或切换知识库
  → 回答
```

本阶段解决“知识在哪里”，暂不追求复杂的跨主题长期研究。

## 2. 用户场景

首批覆盖：

1. 用户不选知识库直接提问。
2. 用户按名称、描述、标签或主题寻找知识库。
3. 多个同名或相似知识库自动消歧。
4. 第一个知识库结果不相关时自动切换。
5. 一个问题需要多个互补知识库时自动组合。
6. 用户预选范围不足时，按范围策略决定是否扩展。
7. 用户指定文档名时，自动使用文档列表和关键词检索。
8. 开放问题先找到最可能相关的资料域，再进入回答或后续探索。

## 3. 产品交付

### 3.1 知识范围模式

前端提供：

- `自动发现`：Agent 可在全部可读知识库中选择。
- `优先使用所选知识库`：先搜所选范围，不足时扩展。
- `仅使用所选知识库`：不得扩展范围。
- `不使用知识库`。

现有知识库选择器和 `@知识库` 继续保留，作为显式偏好或限制。

### 3.2 过程展示

以简洁状态展示：

- 正在寻找相关知识库。
- 已选择的知识库及简短原因。
- 因结果不足而增加或切换的知识库。
- 最终使用的知识库和文档。

不直接展示模型内部思维，只展示可验证的工具行为和范围变化。

### 3.3 用户控制

回答后允许用户：

- 一键排除某个知识库并重试。
- 限定只使用当前知识库。
- 增加一个知识库继续搜索。
- 查看每条引用来自哪个知识库和文档。

## 4. 技术交付

### 4.1 知识库画像

为每个知识库维护可检索画像：

```json
{
  "knowledge_base_id": "ds_xxx",
  "display_name": "财务制度库",
  "description": "公司费用、预算、采购和报销制度",
  "topics": ["差旅", "报销", "预算", "采购"],
  "tags": ["财务", "制度"],
  "representative_documents": [],
  "document_types": ["PDF", "DOCX"],
  "data_sources": ["manual", "feishu"],
  "languages": ["zh-CN"],
  "document_count": 328,
  "last_content_update_at": "...",
  "availability": "ready"
}
```

画像来源：

- 用户填写的名称、描述和标签。
- 文档标题和目录。
- 自动生成的知识库主题摘要。
- 文档类型、来源、语言和更新时间统计。
- 代表性文档和高频主题。

知识库新增、删除或更新文档后增量刷新画像。

### 4.2 知识路由工具

新增或明确拆分以下接口：

```text
search_knowledge_spaces(query, filters, top_k)
describe_knowledge_space(knowledge_base_id)
list_knowledge_space_documents(knowledge_base_ids, query, order_by)
```

`search_knowledge_spaces` 返回：

- 知识库 ID、展示名和摘要。
- 语义相关度。
- 命中的主题、标签或代表文档。
- 推荐原因。
- 更新时间、文档数量和可用状态。

目录搜索只负责找候选知识空间，不返回正文证据。

### 4.3 结构化检索范围

每轮 Agent 保存：

```json
{
  "mode": "discover",
  "preferred_kb_ids": [],
  "active_kb_ids": [],
  "excluded_kb_ids": [],
  "candidates": [],
  "selection_reason": {},
  "routing_confidence": 0.0
}
```

该状态：

- 由工具运行时维护，不依赖自然语言历史重建。
- 在同一会话后续轮次中可继承。
- 可传递给子 Agent，但子 Agent 不能突破父任务范围策略。
- 用户修改范围后立即更新。

### 4.4 最小自主检索循环

1. 从问题提取一个或多个检索意图。
2. 对每个意图搜索知识库目录。
3. 选择 1～3 个候选知识库。
4. 根据问题类型选择语义搜索、关键词搜索或文档列表。
5. 判断结果：
   - 是否与问题相关。
   - 是否包含直接证据。
   - 是否需要上下文扩展。
   - 是否需要另一知识库补充。
6. 结果为空或明显不相关时改写 query 或切换候选。
7. 证据满足回答需求后停止。

默认预算：

- 每个意图最多尝试 3 个知识库。
- 最多进行 2 次 query 改写。
- 默认先搜索 Top-1，证据不足再扩展到 Top-3。
- 用户选择“仅使用”时禁止目录扩展。

### 4.5 检索策略选择

- 指定文件名和短语：优先关键词搜索。
- 开放问题：优先语义搜索。
- 需要了解某库内容组成：先列文档或聚合统计。
- 命中片段语义不完整：扩展父节点或窗口。
- 包含图片、图表问题：同时使用图片检索分支。
- 多个独立意图：分别调用搜索，不拼成一个 query。

### 4.6 可观测性

Trace 记录：

- 路由输入和候选知识库。
- 候选分数和匹配信号。
- 最终选择及选择原因。
- 每次 query、过滤条件和命中数量。
- query 改写、范围增加或切换的原因。
- 搜索停止原因。
- 最终使用的引用。

## 5. 实施任务

### 5.1 Knowledge Catalog

- 定义知识库画像 schema。
- 实现画像全量生成和增量更新。
- 建立知识库画像的语义索引。
- 支持名称、标签、主题、来源、语言和更新时间过滤。
- 为无描述、空库、处理中和失败知识库定义降级行为。

### 5.2 Agent Runtime

- 将目录发现与内容检索拆为不同工具。
- 增加结构化知识范围状态。
- 实现检索结果相关性与最小充分度判断。
- 实现 query 改写和候选切换。
- 统一主 Agent、子 Agent、定时任务和插件中的范围继承。

### 5.3 Frontend

- 增加四种知识范围模式。
- 展示 Agent 当前使用范围和范围变化。
- 支持排除、锁定、扩展和重试。
- 在引用 UI 中展示知识库来源。

### 5.4 Evaluation

- 建立知识库路由离线评测集。
- 增加无前端 `kb_id` 的端到端 Agent 测试。
- 覆盖相似库、同名库、空库、多库互补和无需 KB 的问题。
- 建立线上路由结果和用户改选反馈。

## 6. 验收指标

- KB 路由 Top-1 准确率不低于 80%。
- KB 路由 Top-3 召回率不低于 95%。
- 无前端预选时，知识类问题成功调用 KB 的比例不低于 90%。
- 因错误知识库导致的无结果率较基线下降 50%。
- 用户手动改选知识库的比例低于 15%。
- 所有回答可回溯最终使用的知识库、query 和引用。
- 目录画像更新后能在约定 SLA 内进入路由索引。

## 7. 完成定义

- 无预选知识库的端到端测试稳定通过。
- 知识库画像可全量构建和增量更新。
- Agent 可以自动选择、组合、切换知识库。
- 用户可以限制并修改 Agent 的知识范围。
- 路由 Trace、固定评测集和线上指标均已建立。
- 路由失败具有明确降级策略，不静默从模型记忆生成知识库结论。

## 8. 非目标

本阶段不建设：

- 完整知识图谱。
- 文档版本和历史时态查询。
- 长时间、多分支研究任务。
- 自动写回知识库。
- 知识变化后的主动通知。
