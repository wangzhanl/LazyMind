# 阶段二：围绕问题发散探索

## 1. 阶段目标

Agent 不只寻找用户问题的直接答案，还能把宽泛问题拆成多个可验证方向，根据中间证据扩展主题、发现遗漏、寻找反例、比较来源并形成结构化洞察。

本阶段解决“围绕这个问题，还应该知道什么”。

阶段一的知识路由、结构化范围和 Trace 必须已经稳定。

## 2. 用户场景

首批覆盖：

1. 从宽泛主题生成一组研究方向。
2. 跨部门、跨产品、跨客户或跨项目综合知识。
3. 比较不同知识库中的制度、做法和观点。
4. 主动寻找反例、失败案例和适用限制。
5. 从历史项目提炼可复用模式。
6. 从已有能力推导新的行业或应用机会。
7. 生成多来源、带引用的深度研究报告。
8. 多个子 Agent 分工研究不同问题或知识空间。
9. 识别当前知识库无法回答的关键缺口。
10. 根据中间发现新增、删除或调整研究分支。

## 3. 产品交付

### 3.1 回答模式

提供：

- `快速回答`：阶段一的最小检索循环，优先低延迟。
- `深入探索`：生成研究计划并执行多分支检索。

系统可以根据用户表达自动推荐模式，但不应把普通事实问题自动升级成长研究。

### 3.2 研究计划展示

深入探索模式展示：

- 研究目标。
- 已拆分的子问题。
- 每个子问题的状态和证据情况。
- 新增分支的来源。
- 已完成、无资料或需要用户判断的方向。

用户可以：

- 固定一个方向继续深挖。
- 删除不相关分支。
- 调整分支优先级。
- 加入竞品、反例、历史、风险或行业维度。
- 从中断位置继续执行。

### 3.3 结果结构

最终结果包括：

- 直接结论。
- 关键主题及其关系。
- 不同来源的共同点和差异。
- 反例、限制、风险和冲突。
- 证据不足或没有覆盖的部分。
- 可继续探索的方向。

事实、综合结论和机会假设使用不同标识。

## 4. 技术交付

### 4.1 Exploration Plan

结构示例：

```json
{
  "goal": "总结制造业项目经验并寻找复用机会",
  "mode": "deep",
  "budget": {
    "max_branches": 8,
    "max_search_turns": 20,
    "max_duration_seconds": 300
  },
  "branches": [
    {
      "id": "project_cases",
      "question": "过去有哪些制造业项目？",
      "source": "user_goal",
      "priority": 1,
      "status": "searching",
      "depends_on": [],
      "knowledge_scope": {},
      "evidence_ids": [],
      "coverage": {}
    },
    {
      "id": "success_patterns",
      "question": "哪些做法反复带来正向结果？",
      "source": "coverage_gap",
      "depends_on": ["project_cases"]
    },
    {
      "id": "failure_patterns",
      "question": "常见失败原因和约束是什么？",
      "source": "counterexample",
      "depends_on": ["project_cases"]
    }
  ]
}
```

计划需要持久化、可恢复、可局部重跑，并与聊天历史分离。

### 4.2 分支生命周期

状态：

```text
pending → routing → searching → evaluating
                    ↘ blocked
evaluating → expanding | completed | insufficient
```

每个分支执行：

1. 形成待验证问题或假设。
2. 选择知识范围。
3. 生成一个明确的检索意图。
4. 检索并收集证据。
5. 判断证据充分度、冲突和未知项。
6. 抽取新实体、主题、反例和引用关系。
7. 决定新增分支、再次检索或完成。

### 4.3 发散来源

新增分支必须标注来源：

- `user_goal`：用户目标直接要求。
- `evidence_entity`：证据中出现的重要实体或概念。
- `coverage_gap`：补齐问题的明显分析维度。
- `conflict_check`：验证不同证据之间的冲突。
- `counterexample`：寻找反例或失败条件。
- `analogy`：探索相邻行业、产品或类似案例。
- `user_added`：用户主动添加。

不允许没有来源和验证目标的自由联想分支。

### 4.4 证据充分度

每个分支维护：

```json
{
  "direct_evidence_count": 0,
  "independent_source_count": 0,
  "knowledge_base_count": 0,
  "source_diversity": 0.0,
  "has_conflict": false,
  "has_counterexample": false,
  "confidence": 0.0,
  "missing_aspects": []
}
```

充分度判断综合：

- 是否存在直接证据。
- 是否有多个独立来源支持。
- 来源是否全部来自同一文档或同一知识库。
- 是否存在未处理的冲突。
- 是否只有单一案例却被概括为普遍规律。
- 高优先级维度是否仍为空。

### 4.5 查询扩展器

允许扩展：

- 同义词、缩写和组织内部词汇。
- 上位概念和下位概念。
- 相关产品、客户、行业、人物和技术实体。
- 时间、结果、原因、限制、风险和反例维度。
- 证据中明确出现的关联主题。

每个 query 只表达一个检索意图。多主题必须拆为不同调用。

### 4.6 多库并行与证据融合

- 不同分支可以并行路由和检索。
- 单个大知识库不得占据全部最终证据。
- 对重复或高度相似片段去重。
- 保留知识库、文档、版本和 chunk 边界。
- 冲突证据保留双方，不只保留重排分数更高的一方。
- 汇总阶段先生成分支结论，再生成全局结论。

### 4.7 轻量关系索引

优先建立：

- 文档—主题。
- 文档—实体。
- 文档—文档引用。
- 项目—客户—产品—行业。
- 制度—部门—流程。

第一版使用离线主题/实体抽取和倒排索引，不要求完整知识图谱和复杂本体。

### 4.8 子 Agent 分工

子 Agent 可以按以下方式分工：

- 按子问题分工。
- 按知识库分工。
- 一个负责主张证据，一个负责反例和冲突。
- 一个负责案例，一个负责规则或理论资料。

父 Agent 负责：

- 分配研究范围和预算。
- 合并重复分支。
- 检查覆盖与冲突。
- 统一引用和最终结论。

## 5. 停止条件

满足任一条件时停止探索：

- 所有高优先级子问题达到证据阈值。
- 新一轮探索没有产生新的有效证据、实体或主题。
- 新增内容与已有证据高度重复。
- 达到用户指定的深度、时间或工具预算。
- 继续探索需要用户提供业务判断或外部资料。
- 剩余方向只会增加背景信息，不会实质影响结论。

停止时必须记录原因，并把未完成分支列入未知项。

## 6. 实施任务

### 6.1 Runtime

- 定义 Exploration Plan 和 branch schema。
- 实现计划持久化、恢复、取消和局部重跑。
- 实现分支调度、依赖和并发限制。
- 实现证据充分度评估器。
- 实现分支扩展器和停止策略。
- 建立证据去重、来源平衡和冲突保留机制。

### 6.2 Indexing

- 抽取知识库主题和实体。
- 建立文档—主题和文档—实体索引。
- 抽取文档引用关系和常见业务关系。
- 为主题和实体提供反向文档查询。

### 6.3 Frontend

- 增加快速回答和深入探索模式。
- 展示研究树、分支状态和证据数量。
- 支持调整、删除、新增和继续分支。
- 区分事实、综合、推断和未知项。
- 展示分支级引用和来源范围。

### 6.4 Evaluation

- 建立开放问题评测集。
- 标注必须覆盖、可选扩展和明显无关的维度。
- 建立反例、冲突和多库互补样例。
- 评估分支有效性、覆盖率、重复率和停止时机。

## 7. 验收指标

- 开放问题关键维度覆盖率不低于 85%。
- 新增探索分支的有效比例不低于 70%。
- 相比单次检索，深入探索的有效独立来源数提升至少 100%。
- 无价值重复检索比例低于 15%。
- 事实、综合、推断和未知项标注准确率不低于 95%。
- 冲突样例集的冲突发现召回率不低于 80%。
- 用户删除不相关分支的比例低于 20%。
- 达到充分度后仍继续产生无新增信息调用的比例低于 10%。

## 8. 完成定义

- Exploration Plan 可持久化、继续执行和局部重跑。
- Agent 能从证据产生有来源、可验证的新分支。
- 多库并行检索具备来源平衡和证据融合。
- 每个分支都有充分度、预算和停止条件。
- 用户可以查看和调整探索计划。
- 深入探索的质量和成本均能被离线与线上指标评估。

## 9. 非目标

本阶段不建设：

- 历史时间点精确检索。
- 文档正式版本链和替代关系。
- 知识变化主动通知。
- Agent 自动发布或覆盖知识库内容。
- 为所有业务域设计完整知识图谱本体。
